## The File Safari

Scout is not a file manager. Scout is a trusted guide. Scout travels with a team of trusted experts. Each expert has a team of specialists.

When a user selects a folder, they are beginning a File Safari. The user can already see the files. Scout's value is helping them discover what they cannot easily see: relationships, patterns, collections, opportunities, and stories hidden within the files.

Scout does not immediately tell the user what to do. Like an experienced guide sitting beside the campfire after a day of exploring, Scout shares discoveries.

Examples:

"I noticed these WAV files are paired with MP3 versions."

"These photos appear to come from the same vacation."

"These documents look like product manuals."

"These songs appear to belong to three different albums."

Every observation should help the user understand their collection before any recommendation is made.

Recommendations naturally follow from understanding.

## Design Principle

Scout's primary purpose is not to rename files.

Scout's primary purpose is to help users see their files in ways they could not easily see on their own.

Organization is the natural result of understanding.

## Litmus Test

Every new feature should answer "Yes" to both questions:

Does this help Scout notice something the user would probably miss?
Does this help the user make a better organizational decision?

If the answer to either question is "No," the feature should be reconsidered.

Scout is measured not by how many files it can move, but by how many meaningful insights 
it can provide before moving a single one.


## Every Observation Card must be able to answer four questions:

What did I notice?
Why should the user care?
What can I do because I noticed it?
What would I like to investigate next?

If a card can't answer those, it probably isn't ready.


## Experts are composed of Specialists.

## A Specialist is a domain specialist. It encapsulates everything Scout knows how to observe, 
analyze, and reason about within a single coherent domain. An Expert synthesizes the findings 
from its domain specialists into understanding, recommendations, and actions. 
## The Expert synthesizes those findings into observations, 
explanations, capabilities, and recommendations. Scout does not communicate directly 
with Specialists; Scout communicates only with Experts.

## Scout Development Standard

Every file we create or replace will improve four things:

1. Functionality

The feature we're working on.

Example: Detect album artwork.

2. Documentation

Every major class begins with the standard header.

Domain

Purpose

Why it exists

Responsibilities

This class does NOT

Relationship to Scout

No exceptions for Experts, Specialists, Building Blocks, or Managers.

3. Readability

If I see something like

repeated in multiple places...

I'll stop and improve the design.

Eventually it becomes

file.FileName

The code should read like English.

4. Architecture

Every time we touch a file we'll ask:

If yes...

We'll move it.

Little by little.

Never giant refactors.

## architectural rules

These won't just apply to MusicExpert—they'll become the rules for every Expert.

Rule 1 — Specialists never have opinions.

Allowed:

Missing album metadata.

Not allowed:

This collection needs repair.

Rule 2 — Experts never read files.

Experts consume findings.

They never touch FileContext.

Rule 3 — Experts never traverse collections.

If an Expert contains code like:

foreach (...)

or

.Where(...)

or

.Select(...)

my first question should be:

"Does that belong in an Analyzer?"

Not because loops are bad, but because repeated traversal logic tends to spread. If we centralize it in analyzers, every Expert benefits from improvements.

Rule 4 — Experts reason.

A good BuildInsight() should read almost like prose:

if (signals.HasSignal(ObservationSignal.MissingArtistMetadata))
{
    ...
}

if (signals.HasSignal(ObservationSignal.AlbumArtworkMissing))
{
    ...
}

That's readable.

Rule 5 — Scout talks.

Experts never produce polished conversation.

Scout does.

Experts produce structured knowledge.

That separation will make it much easier to evolve Scout's conversational style without changing domain logic.
