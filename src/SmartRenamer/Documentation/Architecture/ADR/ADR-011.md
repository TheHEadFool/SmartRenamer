
ADR-011: Introduce the Expedition Plugin Architecture

Scout Core owns behavior. Expeditions own experience. Experts own expertise.

Status

Accepted

Context

Scout has evolved beyond a single application theme. We want to allow the user experience to be customized without affecting the application's behavior or reliability.

At the same time, Scout's intelligence must remain extensible through Experts without allowing appearance plugins to modify business logic.

Decision

Scout will support two distinct plugin types:

Expedition Plugins

Expeditions define the visual and experiential aspects of Scout.

They may contribute:

Resource dictionaries
Colors
Typography
Icons
Images
Sounds
Animations
Approved UI controls (through defined extension points)
Metadata via Expedition.json

They may not:

Execute business logic
Access files directly
Modify Expert behavior
Register commands
Replace ViewModels or services
Expert Plugins

Experts extend Scout's observation capabilities.

They may contribute:

New observation logic
Findings
Signals
Recommendations
File type support

They do not control presentation.

Consequences

Benefits

Complete visual customization without risking application stability.
Stable extension points for third-party Expedition creators.
Clear separation between presentation and intelligence.
Safari becomes the reference implementation for future Expeditions.

Trade-offs

Scout Core must provide well-defined extension points.
Some UI flexibility is intentionally limited to preserve compatibility and maintainability.
Initial implementation
Safari ships as the built-in Expedition.
Scout loads Expeditions through an ExpeditionManager.
If no Expedition is configured or loading fails, Scout falls back to the bundled Safari Expedition.
